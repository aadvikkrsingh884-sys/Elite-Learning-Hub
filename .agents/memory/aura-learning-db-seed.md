## Critical: no transaction wrapper
The seed runs WITHOUT `db.transaction()`. Early versions wrapped everything in a single transaction, but the `batchInsert` helper uses the module-level `db`, not the transaction-scoped `db` passed to the callback. The result was that batch inserts silently ran outside the transaction and were killed by the executor timeout before committing. Remove any transaction wrapper from the seed ‚Äî rely on TRUNCATE + CASCADE at the start to ensure consistency.

**Why:** Postgres transactions scope the `db` object passed to the callback, but helper functions that close over the module-level `db` bypass the transaction. At 7,700+ rows per table the executor timeout (300 s) would cut the run mid-stream if everything was wrapped in a long-lived transaction.

**How to apply:** If you add a new seed block, always call `db.insert(...).values(...)` using the module-level `db` (imported from `./index`), never inside a `db.transaction()` callback with a shadowing `db` parameter.
